# Contributing to Sriracha

Thanks for your interest in contributing! Sriracha is **experimental, pre-1.0
software** — the API may change without notice. See [`README.md`](README.md)
for what the library does and where it is in its lifecycle.

## Code of Conduct

This project follows the [Contributor Covenant Code of Conduct](CODE_OF_CONDUCT.md).
By participating, you agree to uphold it.

## Before you start

- **Open an issue first** for anything non-trivial: bug reports, feature
  ideas, or changes to the normalization or tokenization pipelines. This
  avoids wasted work if the direction needs adjusting.
- Trivial fixes (typos, doc tweaks, obvious one-line bug fixes) can go
  straight to a PR.
- **Security issues:** do **not** file a public issue or PR. Follow
  [`SECURITY.md`](SECURITY.md) instead.

## Development setup

Sriracha targets **Go 1.24+**. The dev workflow is driven by
[Task](https://taskfile.dev/); install it once, then everything else is
discoverable from [`Taskfile.yml`](Taskfile.yml).

Common targets:

| Command              | What it does                                                             |
|----------------------|--------------------------------------------------------------------------|
| `task test`          | Run all unit tests                                                       |
| `task test:race`     | Run tests with the race detector (needs CGO)                             |
| `task cover`         | Run tests with coverage and print a per-function summary                 |
| `task lint`          | Run `golangci-lint` (config in [`.golangci.yml`](.golangci.yml))         |
| `task vet`           | Run `go vet`                                                             |
| `task vulncheck`     | Run `govulncheck`                                                        |
| `task fuzz`          | Run a fuzz target (e.g. `task fuzz -- -fuzz=FuzzNormalize ./normalize/`) |
| `task bench`         | Run micro-benchmarks                                                     |
| `task bench:quality` | Run the OpenSanctions / FEBRL4 / NCVR quality benchmarks                 |
| `task generate`      | Regenerate mocks (mockery v3)                                            |
| `task ci`            | Full local CI suite: `test`, `vet`, `lint`, `vulncheck`                  |

A pre-commit hook lives at [`.githooks/pre-commit`](.githooks/pre-commit) and
runs `go test ./...` plus `golangci-lint run`. Enable it once with:

```bash
git config core.hooksPath .githooks
```

## Coding rules

These are hard rules — PRs that break them won't be merged without a very
good reason.

- **No `panic`** anywhere except `MustParsePath`, which is reserved for
  package-level `var` declarations of field paths.
- **All error paths return `error`.** Bounds checks, type assertions, I/O —
  everything that can fail returns an error. Do not use `panic` as an error
  channel.
- **No section-comment separators** (e.g. `// --- Section name ---`).
- Keep external runtime dependencies minimal. If a PR adds one, justify it
  in the description.
- Public API changes should be called out in the PR description. Pre-1.0
  breakage is allowed but should be deliberate.
- User-facing changes (features, fixes, breakage) should be summarised in
  the `[Unreleased]` section of [`CHANGELOG.md`](CHANGELOG.md).

## Testing expectations

- Use [`stretchr/testify`](https://github.com/stretchr/testify): `require`
  for fatal checks, `assert` for non-fatal value checks.
- `t.Parallel()` must be the **first statement** of every top-level test and
  every `t.Run` subtest.
- Prefer table-driven tests with named subtests when several cases exercise
  the same function.
- Do not use ad-hoc mocks. The mocks in [`./mock`](./mock) are generated by
  mockery — after changing a mocked interface, run `task generate` and
  commit the diff. CI's `generated` job fails on uncommitted changes.
- Target **100% coverage** for new code. CI uploads coverage to Codecov.
- For new packages, add micro-benchmarks and at least one fuzz target where
  the package parses or accepts external input. The existing `normalize/`
  and `token/` packages are good references.

## Commit messages

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <short imperative summary>

<optional body explaining the why>
```

Common types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `perf`.
Keep the subject line ≤ 72 characters. The body is optional but encouraged
for the *why* behind a change.

## Pull request flow

1. Fork the repo and create a topic branch off `main`.
2. Make your changes, run `task ci` locally, and push.
3. Open a PR against `main`. **The PR title must follow Conventional Commits**,
   because PRs are **squash-merged** and the title becomes the commit subject
   on `main`. Individual commits inside your branch don't need to be tidy.
4. CI must be green: tests on Linux + macOS + Windows, lint, generated-files
   check, govulncheck, and fuzz.
5. Keep PRs focused — split unrelated changes into separate PRs.
